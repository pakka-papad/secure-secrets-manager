# Generating Master Keys

This guide explains how to generate the high-entropy symmetric keys required to protect the application's Data Encryption Keys (DEKs). These keys act as the **Root of Trust** for the system.

## General Requirements

- **Encoding**: All keys must be provided as **single-line Base64 strings**.
- **Source**: Keys must be high-entropy random bytes generated by a cryptographically secure pseudo-random number generator (CSPRNG).
- **Naming**: Keys are injected via environment variables using the pattern `MASTER_KEY__VX` (e.g., `MASTER_KEY__V1`, `MASTER_KEY__V2`).

---

## Supported Algorithms

### 1. AES-256-GCM
This is the default and recommended algorithm for Master Keys. It requires exactly **32 bytes** (256 bits) of random data.

#### Generation using OpenSSL (Recommended)
```bash
openssl rand -base64 32 | tr -d '\n' > master_key_v1.txt
```

#### Generation using /dev/urandom (Linux/macOS)
```bash
head -c 32 /dev/urandom | base64 | tr -d '\n' > master_key_v1.txt
```

**Explanation of Parameters:**
*   `openssl rand`: Generates cryptographically strong pseudo-random bytes.
*   `-base64 32`: Generates 32 bytes and encodes them in Base64.
*   `tr -d '\n'`: (Critical) Removes any trailing newlines. The application expects a single, continuous string.

### 2. ChaCha20-Poly1305
An alternative high-performance software-based AEAD algorithm. It also requires exactly **32 bytes** (256 bits) of random data.  
Follow the same steps from AES-256-GCM section to generate the keys.

## Configuration

Once you have generated your key, export it to your environment.

### Linux / macOS / Bash
```bash
export MASTER_KEY__V1=$(cat master_key_v1.txt)
export MASTER_KEY_DEFAULT_ALGORITHM=AES-256-GCM # Or CHACHA20-POLY1305
```

## Best Practices

1.  **Rotation**: When adding a new key (e.g., `V2`), keep the old key (`V1`) in the environment. The system will automatically detect the new version, promote it to `ACTIVE` for new secrets, and keep the old one as `RETIRED` for decrypting existing data.
2.  **No Whitespace**: Always ensure there are no spaces or invisible characters in your environment variable value. A common cause of `IllegalStateException` is an accidental newline at the end of the Base64 string.
3.  **Storage**: Never store these keys in your source code or `application.yml`. Use a secure vault, an environment variable manager, or a hardware security module (HSM).

---

<a name="compromise-policy"></a>
> [!WARNING]
> **COMPROMISE POLICY**: If you suspect a Master Key has been leaked, you must mark it as `COMPROMISED` immediately. Note that all secrets protected by a compromised key will be rendered unreadable to protect against data draining. You must replace those secrets with fresh values.
